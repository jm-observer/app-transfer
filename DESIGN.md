# 项目设计需求文档

---

### 1. 场景

| 编号 | 描述 |
|------|------|
| 1.1 | **程序替换**：在部署或调试阶段，需要用一个自定义的 “中转程序” 替换掉原有的可执行文件（以下简称 *原始程序*），以便拦截其 I/O 流。 |
| 1.2 | **启动方式**：另一个进程（称为 *启动进程*）通过标准输入、标准输出、标准错误流来启动 *中转程序*。启动时 *中转程序* 会在同一目录下解析出需要被替换的 *原始程序* 名称（该名称在编译时硬编码进二进制），随后启动该 *原始程序*。 |
| 1.3 | **消息转发**：*中转程序* 将 *启动进程* 与 *原始程序* 之间的所有 stdin/stdout/stderr 消息进行透明转发，使两者的交互行为保持不变。 |
| 1.4 | **日志记录**：所有转发的数据（包括 stdin、stdout、stderr）必须同步写入同一目录下的日志文件。日志文件名在编译阶段通过宏或构建脚本注入到二进制中。 |
| 1.5 | **部署约束**：中转程序、原始程序以及日志文件全部位于同一文件夹，便于复制、打包或容器化部署。 |

### 2. 目标实现

| 目标编号 | 具体实现要点 |
|----------|--------------|
| 2.1 **编译时绑定替换目标** | - 在 `Cargo.toml`（或对应的构建脚本）中定义两个编译时常量：<br> 1. `TARGET_EXE_NAME` – 需要被替换的原始可执行文件名（如 `original.exe`）<br> 2. `LOG_FILE_NAME` – 日志文件名（如 `transfer.log`）<br> - 使用 `build.rs` 或 `env!` 宏将上述常量硬编码进二进制。 |
| 2.2 **启动并托管子进程** | - 程序入口首先解析当前工作目录，构造 `<current_dir>/<TARGET_EXE_NAME>` 的完整路径。<br>- 使用 Rust 标准库 `std::process::Command`（或等价语言的 API）以 **inherit** 方式启动子进程，保留子进程的 stdin、stdout、stderr 句柄。 |
| 2.3 **I/O 双向转发** | - 为每个方向（stdin→子进程、子进程→stdout、子进程→stderr）建立独立的异步任务或线程。<br>- 使用 `tokio::io::copy` / `async_std::io::copy`（或同步 `std::io::copy`）在两个流之间转发数据。 |
| 2.4 **日志写入** | - 为每条转发的数据块添加时间戳与方向标记（`[STDIN]`、`[STDOUT]`、`[STDERR]`）。<br>- 采用追加模式打开 `<current_dir>/<LOG_FILE_NAME>`，使用缓冲写入以降低磁盘 I/O 开销。 |
| 2.5 **错误与退出处理** | - 若子进程异常退出，记录退出码并将相应信息写入日志。<br>- 中转程序本身在子进程结束后返回子进程的退出码给启动进程。 |
| 2.6 **跨平台兼容** | - 使用统一的路径拼接（`PathBuf::join`）避免 Windows 与类 Unix 路径差异。<br>- 在 Windows 平台下确保使用 `\` 作为路径分隔符（Rust `std::path` 已自行处理）。 |
| 2.7 **构建与发行** | - 提供 `cargo build --release`（或等价）生成单文件可执行体。<br>- 打包脚本 (`zip`/`tar`) 只需把 `target/release/<binary>`、`original.exe`（若需要）和 `README.md` 放入同一目录即可。 |

### 3. 测试

| 测试编号 | 测试目标 | 步骤概述 | 预期结果 |
|----------|----------|----------|----------|
| 3.1 **编译常量注入** | 验证二进制包含正确的 `TARGET_EXE_NAME` 与 `LOG_FILE_NAME` | 1. 编写 `build.rs` 将常量写入 `env!("TARGET_EXE_NAME")` 等。<br>2. 编译后使用 `strings` 或运行时打印常量。 | 常量值与 `Cargo.toml` 中配置一致。 |
| 3.2 **子进程启动** | 确认中转程序成功启动目标可执行文件 | 1. 在同目录放置 `original.exe`（或任意可执行脚本）。<br>2. 运行中转程序。<br>3. 检查子进程是否在任务管理器/`ps` 中出现。 | 子进程 PID 可见，且其工作目录为当前文件夹。 |
| 3.3 **I/O 透明转发** | 检验 stdin→子进程、stdout/stderr→父进程的完整转发 | 1. 向中转程序的 stdin 发送 `"hello\n"`。<br>2. 子进程（例如 `cat`）回显相同内容。<br>3. 捕获父进程的 stdout，确认收到 `"hello"`。 | 输出完全一致，未出现丢帧或字符乱码。 |
| 3.4 **日志写入完整性** | 确认所有转发数据均写入日志文件 | 1. 完成 3.3 的交互后，打开 `transfer.log`。<br>2. 检查是否有 `[STDIN] hello`、`[STDOUT] hello` 两行记录，且时间戳递增。 | 日志文件存在，记录完整且顺序正确。 |
| 3.5 **异常退出处理** | 子进程异常退出（非 0 码）时，中转程序应返回相同退出码并记录日志 | 1. 使用 `cmd /C exit 3` 作为目标可执行文件。<br>2. 运行中转程序。<br>3. 检查父进程返回值、日志文件中是否有 “exit code: 3”。 | 父进程返回 3，日志记录对应信息。 |
| 3.6 **跨平台路径解析** | 在 Windows 环境下确认路径拼接使用 `\`，不产生非法路径 | 1. 在 `C:\test\transfer\` 目录下运行程序。<br>2. 程序打印解析后的子进程完整路径。 | 输出形如 `C:\test\transfer\original.exe`，路径有效。 |
| 3.7 **并发转发性能** | 大量数据（如 10 MB）转发时不出现阻塞或数据丢失 | 1. 使用 `dd if=/dev/urandom bs=1M count=10` 生成 10 MB 随机流。<br>2. 将其 pipe 给中转程序，子进程使用 `cat` 直接输出。<br>3. 比对原始与最终输出的 MD5。 | 两端 MD5 一致，转发完成时间在可接受范围（≤ 5 s）。 |

**测试环境**：Windows 10/11（MSYS2 Bash），Rust 1.74（或对应语言最新稳定版），`cargo test` 脚本可自动执行 3.1‑3.5。

### 附录

- **关键配置示例（Cargo.toml）**

```toml
[package]
name = "io_forwarder"
version = "0.1.0"
edition = "2021"
build = "build.rs"

[dependencies]
tokio = { version = "1", features = ["full"] }   # 异步 I/O
```

- **build.rs 示例**

```rust
fn main() {
    println!("cargo:rustc-env=TARGET_EXE_NAME=original.exe");
    println!("cargo:rustc-env=LOG_FILE_NAME=transfer.log");
}
```

- **运行示例**

```bash
# 假设二进制名为 io_forwarder.exe
> io_forwarder.exe
# 交互后可在同目录查看 transfer.log
```
